Workflow Synopsis

From Kickoff to Delivery

Banner development is a production discipline as much as it is a technical one. This synopsis covers the full process — from the first kickoff meeting through final asset delivery — including how I manage creative direction, asset preparation, animation build, cross-size adaptation, staging, compliance, and trafficking. It's the same process I've run across every agency and every campaign for over 11 years.

Kickoff & Planning

Every campaign begins with a kickoff meeting where I align with the creative and production teams on animation treatment, brand tone, hard deadlines, master banner size, resize requirements, and platform-specific standards. I receive creative PSD files for the master banner size — typically a 300x250 — along with a PDF copy of the brief that I reference throughout development for animation direction and brand guidelines.

Development — Master Banner

I start every project from a custom boilerplate template matched to the animation style the campaign calls for. Because I work with recurring brands, I maintain templates tailored to specific creative directions that accelerate production from the jump. My custom GreenSock framework (TweenMax/TimelineMax) provides pre-built animation methods and functions, timeline manipulation, asset preloading, a scrubber for playing, pausing and scrubbing the animation timeline, and debugging scripts for cross-browser troubleshooting.

I create a new PSD file for every banner size, duplicating image assets from the creative PSD handed off to me. All elements are exported individually as JPG or PNG at retina resolution — occasionally SVG when vector assets are involved, which I manipulate in Illustrator if needed. I then use code to scale retina-sized assets back down to banner dimensions, retaining clarity and high resolution across standard and retina-enabled devices.

Once assets are exported, I build the banner structure — creating div elements for each image asset, positioning them using GreenSock, and setting starting opacities, positions, and pre-animation states. I then script any custom animation timeline functions the project requires beyond what my framework already provides, and begin constructing the master timeline by chaining and offsetting timeline functions together based on the creative direction. I preview and adjust in Chrome using DevTools throughout the entire build, refining timing, easing, and positioning until the animation matches the intended feel. I verify the banner meets the project's time constraints using my scrubber tool and confirm compliance with IAB and browser standards before delivering.

Delivery & Staging

Once the master banner is approved, I use it as the foundation for all remaining sizes, which significantly reduces development time per unit and is a core reason I can manage multiple campaigns simultaneously. I adapt animation and layout as needed across sizes — the same animation does not always translate identically at different dimensions, so I make subtle creative decisions to preserve the brand's voice and the creative team's intent across every format.

Before final delivery, I compress all image assets to bring each banner's zipped file weight below platform restrictions, verify clicktag functionality, and run all units in current browsers to catch any rendering issues. I then stage the full campaign on a custom PHP-based staging site that I built and maintain as a template. Each campaign gets its own staging instance populated with banner sizes, animation timing, kickoff date, backup images, and brand details. Project managers use the staging site to preview and pre-approve banners before sending the viewing link to clients for final sign-off. The staging site also provides direct download links for zipped banner files per size, backup images for all units, and a link to validate all banners against Google's HTML5 banner validation tool for standards compliance.

Once the client fully approves the campaign, project managers download all packaged assets for trafficking to the final ad-serving platform — most commonly DoubleClick, for which I maintain verified enabler scripts in my boilerplate template. For campaigns requiring Flashtalking, I have a separate template built to that platform's specifications.

Scale & Output

Small campaigns typically involve one creative direction with around four banner sizes. Larger campaigns can include up to five versioned concepts across two or three creative directions, each with up to six sizes. I routinely juggle two to three campaigns simultaneously and have delivered as many as 50 banner units in a single week.

Workflow Tools

Banner Staging & Review Site

Production doesn't end when the banners are built. Getting them reviewed, approved, and delivered cleanly requires just as much care as the code itself. This staging site is a custom-built internal tool I developed to manage that entire handoff process — giving art directors, project managers, and clients a single organized destination to view live banners, track status, and pull what they need without back-and-forth emails or shared drives.

Each project gets its own instance of the site, configured per campaign. Banners are organized by IAB size, displayed as live-running units with fallback image thumbnails, and paired with download links for source files and a modal lightbox for full-size preview. A manifest-driven file drawer exposes every asset inside each banner package. Status badges track each unit from Draft through In Review, Approved, and Delivered. A linked report page auto-generates a full project summary — banner counts, file totals, runtimes, and a delivery timeline — from the same data, with a clean print-to-PDF output for records.
